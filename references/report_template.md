## 写作要求

### 整个文档必须包含

1. **整体结构**
2. **目录**
3. **Feature 标题**
4. **总结**
5. **技术趋势**
6. **需持续关注**

### 每个 Feature 必须包含

1. **背景**：为什么要做这个改动，现有机制有什么局限
2. **解决的问题**：具体问题列表，尽量量化
3. **如何做**：技术实现，含关键函数/接口/数据结构，必要时附代码片段
4. **收益**：改动的实际价值，优先引用 cover letter 中的 benchmark 数据

### 深度要求

- **所有 feature 都要详细写**，每个约 500 字以上
- 有 benchmark 数据的 feature 必须引用数据
- 性能数据的判定：完整阅读封面信 + 所有 patch 提交日志正文。以下形式均算：
  - benchmark 表格（kernbench、fio、sysbench、perf bench 等）
  - bpftrace / perf / ftrace 观测数据
  - 时间对比（"从 X 秒降至 Y 秒"）
  - 内存/CPU 节省量
  - 百分比改善
  - 作者在正文任意位置提到的任何量化数字
- 有内核崩溃 / syzbot 复现的 bug fix 要描述复现路径
- 引用作者原文中的关键句子（用引号）

### 术语规范

- 保留各子系统的英文技术术语（见 `subsystems.md` 各子系统的"保留英文术语"列表）
- 中文说明后第一次出现时括号注英文：透明大页（THP）
- 代码、函数名、文件名用 `代码格式`
