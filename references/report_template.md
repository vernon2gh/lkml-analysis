# 报告格式模板

## 整体结构

报告顶部：

    # Linux <子系统名> 最新 Feature 分析报告

    **生成时间：** YYYY-MM-DD
    **时间范围：** 最近N天
    **数据来源：** <邮件列表名>

目录区：

    ## 目录
    1. [Feature 标题](#1-Feature-标题)
    2. [下一个 Feature](#2-下一个-Feature)

Feature 区块：

    ## 1. Feature 标题

    **系列**：`[PATCH vN 00/N] xxx`
    **作者：** xxx
    **版本：** vN（N个patch）/ RFC

    ### 背景
    [为什么需要这个改动？历史背景、现有机制的局限性]

    ### 解决的问题
    - 问题1
    - 问题2

    ### 如何做
    [技术实现细节，包含关键函数/接口/数据结构变化，必要时内联关键代码片段]

    ### 收益
    [有数据：直接列出数字和测试环境]
    [无数据：「作者未提供性能数据，[从代码逻辑推断的预期收益]」]

总结区（四个分类表格）、技术趋势、需持续关注——详见下方写作要求。

---

## 写作要求

### 整个文档必须包含

1. **整体结构**
2. **目录**
3. **Feature 标题**
4. **总结**
5. **技术趋势**
6. **需持续关注**

### 总结区格式

本次分析摘要，按类型汇总本期所有入选 feature，方便快速浏览。每行包含编号、系列名（版本）、核心要点或量化数据。分四类：

**新机制 / 新接口**

| # | 系列 | 要点 |
|---|------|------|
| N | **名称** [vX, Np] | 一句话描述 |

**性能优化**

| # | 系列 | 量化数据 |
|---|------|---------|
| N | **名称** [vX, Np] | 有数据则填数字，无则写预期收益 |

**Bug Fix**

| # | 系列 | 影响 |
|---|------|------|
| N | **名称** [vX] | 修复内容及 Fixes 标签 |

**内部优化 / 清理**

| # | 系列 | 要点 |
|---|------|------|
| N | **名称** [vX, Np] | 一句话描述 |

### 每个 Feature 必须包含

1. **背景**：为什么要做这个改动，现有机制有什么局限
2. **解决的问题**：具体问题列表，尽量量化
3. **如何做**：技术实现，含关键函数/接口/数据结构，必要时附代码片段
4. **收益**：改动的实际价值，优先引用 cover letter 中的 benchmark 数据

### 深度要求

- **所有 feature 都要详细写**，每个约 500 字以上
- 有 benchmark 数据的 feature 必须引用数据
- 性能数据的判定：完整阅读封面信 + 所有 patch 提交日志正文。以下形式均算：
  - benchmark 表格（kernbench、fio、sysbench、perf bench 等）
  - bpftrace / perf / ftrace 观测数据
  - 时间对比（"从 X 秒降至 Y 秒"）
  - 内存/CPU 节省量
  - 百分比改善
  - 作者在正文任意位置提到的任何量化数字
- 有内核崩溃 / syzbot 复现的 bug fix 要描述复现路径
- 引用作者原文中的关键句子（用引号）

### 术语规范

- 保留各子系统的英文技术术语（见本文件末尾"子系统参考"各节的保留英文术语列表）
- 中文说明后第一次出现时括号注英文：透明大页（THP）
- 代码、函数名、文件名用 `代码格式`
